参考linux的一些特殊符号，主要提两个，第一个是管道符(|)另一个是连接符(;)。他们两个都能连接n个命令并执行。不同点在于管道符会把前面的命令的返回值传给后面的命令作为参数，并把最后一个命令的返回值作为总的返回值；而连接符不存在返回值，只是依次执行命令。

因此我们直接127.0.0.1|ls /发现flag文件，直接127.0.0.1|cat /flag就得到flag

这里解释两个东西

第一个问题

首先是关于空格过滤用到的$IFS，$IFS是shell里的环境变量用于内部字段分隔符，可以设置分隔符为某个符号，默认值为空格、制表、空行。其中$IFS$9中的$9指的是传过来的第九个参数，只要不为0（0无意义）就会返回分隔符也就是空格。他没加{}的原因在于两个$已经足够保证能确定出$IFS了，而使用${IFS}则是为了防止与以IFS开头的变量混淆。

第二个问题

这道题和上道题不同，上道题既能用管道符也能用连接符，但这个题前面可以用管道符，最后一步用管道符?ip=127.0.0.1|w=g|cat$IFS$1fla$w.php却什么结果都得不到了，甚至?ip=127.0.0.1|w=g;cat$IFS$1fla$w.php也得不到结果，这是为什么？

很简单，因为代码审计看到最后用的是print而不是include，print会将$a打印出来，而$a是什么？是我们得到的输入命令的返回值。只要我们使用管道符，相当于最后print(返回值)，中间的各个过程它内部实现后就挂掉了（相当于给黑盒封装了），只给我们最后一个返回值，而注释不算做返回值自然什么都读不到。而用连接符，虽然最后返回值也得不到flag，但是中间的过程不是被黑盒封起来的，我们自然可以获得里面的注释。

文件上传漏洞很多都是写一句话木马然后蚁剑，难点在于如何绕过过滤

首先，网页可能对文件格式进行过滤，这种一般都是根据Content-type判断的，可以通过修改Content-type绕过

其次，网页可能对文件的名字进行过滤，比如检测到名字带有php时就禁止上传，这种可以通过更改为php,php3,php4,php5,phtml,pht等尝试绕过

PS：对于服务器而言，他检测你的文件类型一定是从后向前找.xxx来判断的，比如你上传一个文件111.php;.jpg，那么他会被当做图片处理。也就是说，你可以直接上传一个php文件然后在burpsuite里修改Content-type绕过、也可以上传一个合法类型文件然后在burpsuite修改后缀为php，这两种本质相同，但一定要保证你最后上传上去的后缀名类型是可以被执行的（比如php）而不是jpg这种不可执行的文件。

关于网关、隧道和中继的理解

这一块篇幅很短，但是感觉讲的很抽象很晦涩。。。也不保证现在的理解就是对的，先写上吧，之后如果发现当下的理解是错的再改吧

1.网关

说起网关就不得不说起代理，这两个东西都在客户端和服务端起作用，在我理解中，他们的区别就在于：代理（反向代理、内容代理、缓存代理。。。）的目的本质在于对同一协议内容的转发，哪怕它会增减原http报文某些内容或者根据某些条件将报文转发到不同的服务器。而网关，更像是一个黑盒的接口，你送进去报文，它会通过网关进行处理送给你http报文所请求的资源，哪怕这个资源不是http报文，而是文件或者图片都可以。

因此之前的那篇文章我才会说，他们的差别在于代理处理同一协议，而网关粘合不同协议，就是这个意思。虽然现在因为很多网关也实现了代理的转发功能、很多代理也实现了网关的不同协议资源交互的功能，但是本质上，他们还是有区别的。

2.隧道

首先看隧道是怎么工作的。首先客户端发送connect请求给网关，要求网关建立起和客户端的隧道连接。彼此验证连接后，隧道就建立起来了（此时隧道建立在客户端和网关之间了）。这时客户端通过隧道发送的所有数据都会被直接被网关处理（即盲转发，比如网关的作用是与服务器进行ssl通信，那么这时客户端发送给网关的数据会被直接发给服务器），同样网关获得的响应也会直接通过http隧道转发给客户端。

这么做最初的目的是通过防火墙传输加密的ssl流量以增强安全性。对于加密的ssl流量（非http流量），防火墙无法识别而会导致无法传输。而如果建立起http隧道，无论什么样的协议我都能送过去，这太方便了，但是这样做的安全问题也就又显露出来了。没有防火墙时内容无法进行内容/协议的过滤、而用了防火墙+隧道后，并没有本质上解决这个问题，只是相比原来想输入什么就输入什么，现在问题多了一步就是如何才能成功建立隧道，安全性的的确确是增加了。

对于http/https网关（这里http/https的含义为客户端与网关用的http，网关与服务端用的https，见P210），实现起来本质是客户端与网关直接进行80端口的简单http连接，网关进行ssl加密后与服务端进行443端口的ssl连接（即https）。这会导致服务端接受来自网关的数据时用的仍然是不安全的http。因此这种方式其实是很out的，我们主要看下面的方式

（1）直接让http报文经过ssl加密后通过专门的ssl连接（443端口）传输【即https】

上面http/https网关问题的本质在于网关虽然可以处理与服务端之间的ssl会话，但是只能与客户端进行http的交流（网关相当于黑盒子，我们可以控制盒子里面怎么进行操作，但输入只能是http报文我们没法改），方法（1）其实就是直接摒弃网关，直接让客户端和服务器进行https连接。对于这种直接的https连接，打从一开始就不是用的http报文，而是通过443端口彼此发送ssl加密的报文后解密。但这样做的弊端在于无法同时使用其他的协议处理除了https以外的内容（比如ftp），相当于这一条连接上客户端能传递出去的信息内容被卡死在https上了，在这一点上甚至比不上上面的http/https网关。

（2）ssl隧道（通过http隧道建立起的ssl连接）通过80端口在http报文中传输ssl流量

方法（2）的思路是“客户端和网关直接不是无法传递ssl加密的信息吗，那我就直接加一个客户端与网关的隧道不就好了”。因此客户端与网关之间不必使用http了，直接用ssl加密后的报文通过一条http隧道连接就完事了，相当于客户端与网关、网关与服务端都实现了https，就完美了（客户端把ssl信息送入隧道后承载于http隧道，http隧道此时可以携带别的协议信息，出隧道时再ssl封装，由网关与服务器的https连接把“打了两次包”的数据给服务器让他解析）。这种方法相比（2）最大的好处就在于，他可以在传递ssl信息的同时借http隧道传递其他协议的内容，不但安全性有保障（隧道的盲转发是不透明的，保证了安全性），而且范围也大多了，再也不必再为了别的协议专门开一个连接了，直接顺带着就传过去了。

3.中继

其实就是不规范版的代理，做的就是盲转发。不规范自然会导致很多问题，之前代理说的哑代理就是一个很好的例子。

关于Connection首部和哑代理的问题

之前第一次看HTTP权威指南的时候这里就有很多地方不理解，第二遍回看的时候发现还是自己看的不够细，书上其实解释的很清楚了

1.为什么哑代理会在与服务器建立keep-alive后挂起客户端的请求？不可以同时发多个请求吗？

A：书上在哑代理这里第一句话就是：问题出在代理上。对于旧版的代理而言，他们不知道有持久连接这回事，他们只会等一个连接结束后再开启一个新连接。在我理解看来，其实本质原因有两点，一是这些代理没有实现并行连接(这种代理本质是单线程的WEB服务器/P125)，二是服务器/客户端认识keep-alive而代理不认识导致的连接关不上。这两个问题只要解决其中一个就可以把这些蠢代理变成聪明代理了。一的解决是对于1.1版本的代理而言的，如果让代理实现并行连接，这样就可以让代理同时承载两个持久连接并处理服务器和客户端之间的HTTP事件；而二的解决是一个约定的标准，通过禁止1.1版本以前的代理转发Connection的方式直接ban掉了持久连接，自然不会有哑代理的问题。

2.HTTP1.0用的是keep-alive，在HTTP1.1中用”默认持久连接，取消时使用close”的方式进行了代替，这样做好像也解决不了哑代理啊。

A：人家这么做不是为了解决哑代理，而是为了优化机制。哑代理的解决不是单纯改持久连接方式就能改的，本质原因还是因为中间的蠢代理。而HTTP1.1也依然延续了上一问说的的解决方式：即对于包括1.1以上版本的代理，已经实现了并行连接就不会有哑代理了；而对这之前版本的代理，直接阉割掉这个功能防止出问题。

作为一个在高考结束后的暑假被自己喜欢的女生推荐而开始逛知乎、最后被上面的精英文化洗脑的大学生，我现在是感到十分羞愧的。因为这些精英文化给当时成绩优异的我一种优越感，为此，我伤害了很多关心我的人，也把自己放进“内卷”的竞争中，直到昨天，我仍然处在绩点不够的焦虑之中，我想在绩点上努力，但是我对它完全不感兴趣。倘若我的理想是未来做学术研究，那我会自发地去争取高绩点，但是对现实中实用主义的我，我不喜欢，我一点也不喜欢为了做题去学知识，一点也不喜欢为了竞争和同伴彼此勾心斗角，一点也不喜欢把自己束缚在无尽的焦虑，一点也不喜欢学我不感兴趣的东西。但是过去的我，哪怕能明确意识到自己未来想做的事，也没法把自己从内卷中拽出来，我害怕啊，我害怕自己不被认可，我害怕事情不朝着我预想的方向发展，我害怕担风险。在这种紧张的气氛中，哪怕是我喜欢学的东西，我也会不自觉的去思考他在这场竞争中能带给我多少筹码，这让我始终没法沉下心来学习安全，因为我害怕学了浪费时间，害怕自己在竞争中因为不够主流而被淘汰。

但是越是在这场竞争中陷得越深，越是无法置身事外。不可能所有人都到达顶点，你初中上的是你们市最好的初中，你还需要去竞争更好的高中，再去竞争更好的大学。在大学之前，很多人觉得自己身边没有那么厚重的“卷”的氛围，这很大程度上是因为无论中考还是高考，你和你身边的同学都是“外卷”，高考几十万人，如此大的盘，导致你身边的人从本质上并没有在和你竞争，你们是一致向外去和外校竞争。而大学，一个专业撑死也就三位数，保研奖学金名额就那么几个，哪怕你完全不想争取这些资源，当你身边有人开始竞争的时候，就会引发恐慌，比起拿奖学金拿保研名额，更多人是害怕无所事事的自己哪一天就被淘汰掉了。

我曾经也坚定的认为，努力绝对是比无所事事要强的，不知道干什么，那就先去做。但是现在我的价值观完全变化了，无所事事的时间恰恰是我认为宝贵的，这段时间是大学本就该拥有的“试错”时间，就是用来让你慢慢找到你想做的东西的，我真的不信有人能大学四年一直玩不会感到无聊的，如果真有，那恐怕游戏就是他想做的事情。然而现在，焦虑一旦被引爆，这段试错的时间就会变成被“大众思想”所鄙视的鄙视链最下游，所有人都被焦虑强行赶到学习的流水线上，哪怕他们不知道自己想学什么，为什么学。

这么学到最后，大概率还是会发现自己处于内卷当中了，因为能在无穷无尽的竞争中生存到最后的人只是极少数，绝大多数人在竞争的某个环节就被淘汰了。而我，就是大学绩点竞争当中的一个失败者。曾经，我深深的恐惧着失败，而如今，我能坦然面对。这是为什么？

当我看到我的专业排名17/51的时候，我有一种难言的挫败感。17/51，这意味着什么，意味着在绩点竞争中的失败，意味着无法保研，意味着曾经的我眼中的“人生完蛋了“。然而，这意味着我没好好学习吗？我之前几个学期，学东西追求“通透”，我不喜欢刷试题，讨厌课上回答问题，不喜欢巴结老师，我更喜欢自己学自己的。但是考的却没巴结学长学姐搞资料、疯狂刷试卷、和老师助教频繁“交流”的人考的好，我来气啊。我思考了一晚上，发现是我自己，被考试绑架了。从小到大，我考试都名列前茅，在考试上，我用自己的努力成为了胜者，因此，我被这种优越感绑架了。我渴望自己能一直考好，渴望始终能俯视身边的人。然而这种心态是病态的，而有着这种心态的也绝不止我一个人，而是所有被考试绑架的学生们，这是当代教育所追求的公平，却也是当代教育的可悲。

我思考了我为什么考不过别人，发现我对学习的价值观和高绩点完全冲突了。在我眼里，高绩点应该等价于兴趣+努力+方法并最终等价于学通，我一直以兴趣作为驱动力，我喜欢的东西我就努力学，不喜欢的东西我就水，我喜欢的课老师哪怕很严我也要去上，我不喜欢的课哪怕老师很水我一样当做空气，这怎么可能拿到高绩点？现代高绩点的公式已经不看兴趣了，他等价于努力+方法，只要你拼命卷，学会高分技巧，不需要学会照样能第一第二，不感兴趣的课照样能学好。

对于曾经被我病态的“精英思想”所伤害过的我的朋友，我真诚的说一句对不起，我希望你之后能找到你的方向，并希望你能减轻一点点大学中的焦虑。

异或注入中字符串的异或-sql的弱类型

今天在bugku上刷到一个叫sql注入2的题。题目过滤了很多保留字，像是and，or，逗号，空格等等。题目用了mid的倒序取反再截取最后一位，再根据uname进行布尔盲注，最后把所有字符再取倒序就得到passwd里的内容。根据特征再md5解密就得到密码。

脚本思路参考网站

但是重点来了，我还看到有用异或进行布尔盲注的(网站)，我感到很惊奇，异或注入返回值不是应该只有数字吗（非数字在^中会被当做0），然而用户名却是’admin’，怎么可能用它进行盲注呢？

我又找了个在线跑sql的网站，发现select ‘admin’^0^’’；明明返回的是0啊，不返回admin那wp里面是怎么用它布尔盲注的??

最后问了学长才知道，这个登录界面后台可能是用的username=uname，我们uname”‘admin’^0^’’”返回0，就是username=0，这样是可以爆出全表的。因为username=0，实际上是查询所有能解析为0的字符串，也就肯定能查到admin啦。

而什么字符串可以解析为0呢？注意sql是弱类型，所以以0开头的非纯数字的字符串、空串还有不包含任何数字的字符串都会被解析为0。而如果一个字符串以数字开头而又不是纯数字，那就会被解析为首个数字（以遇到非数字为终止）。（PS:思考一下，一直’abc123’=0成立，’abc’=0成立，为什么’abc123’=’abc’不成立？A:因为前两者的成立是建立在将字符串作为数字解析的前提下，而后者的两个字符串都被当做字符串解析，肯定就不一样了啊。）

文件包含漏洞和php伪协议

最近在刷Bugku的题，看到题目”flag在index里”用到了文件包含漏洞和php伪协议的知识，想到之前也做过有关这个知识点的题，但是当时没有仔细学，这次打算仔细学学这些内容。

0.文件包含漏洞

PHP中文件包含函数有require()/require_once()/include()/include_once()，会将参数包含（使当前代码类也能调用参数所在文件内的类）并当做PHP代码运行，而include()运行过程如果出错，抛出一个警告之后仍会继续运行。而如果用户包含一些恶意文件呢？就会执行一些非预期代码，这就是文件包含漏洞。

1.本地文件包含漏洞

(1)无限制文件包含漏洞

可以用来读取一些系统默认的敏感文件

Windows：

c:\boot.ini // 查看系统版本

c:\windows\system32\inetsrv\MetaBase.xml // IIS配置文件

c:\windows\repair\sam // 存储Windows系统初次安装的密码

c:\ProgramFiles\mysql\my.ini // MySQL配置

c:\ProgramFiles\mysql\data\mysql\user.MYD // MySQL root密码

c:\windows\php.ini // php 配置信息

Linux：

/etc/passwd // 账户信息

/etc/shadow // 账户密码文件

/usr/local/app/apache2/conf/httpd.conf // Apache2默认配置文件

/usr/local/app/apache2/conf/extra/httpd-vhost.conf // 虚拟网站配置

/usr/local/app/php5/lib/php.ini // PHP相关配置

/etc/httpd/conf/httpd.conf // Apache配置文件

/etc/my.conf // mysql 配置文件

(2)session文件包含漏洞

Q：Session的文件名是什么？

A：”sess”_+sessionid

利用条件：

(a).既然属于文件包含漏洞，肯定最起码的得有个include()能让我们执行自己的文件吧，也就是要有1.(1)的先决条件

(b)session可控，能让我们传入恶意代码

满足了以上条件，就可以利用这个漏洞了

magic_quotes_gpc函数在php中的作用是判断解析用户提示的数据并对某些字符进行转义。在magic_quotes_gpc=ON的时候，如果有get、post、cookie过来的数据中有单引号（’）、双引号（”）、反斜线（\）与 NUL（NULL 字符），就会自动给他们添加反斜线（\）来进行转义。如果为Off还想保留这个功能，就需要自己调用addslashes这个函数。

利用条件：

magic_quotes_gpc=OFF且php版本<5.3.4（php5.3.4已经解决了%00的漏洞。顺带一提，php5.3之后set_magic_quotes_gpc就没有了…甚至php6就没有这个特性了QAQ，时代变了呀）

2.远程文件包含漏洞

如果PHP的配置文件allow_url_fopen和allow_url_include设置为ON，include/require等包含函数可以加载远程文件，如果远程文件没经过严格的过滤，导致了执行恶意文件的代码，这就是远程文件包含漏洞。相比本地文件包含漏洞，远程文件包含漏洞危害更大，因为普通的本地包含漏洞只能读取网页内部的本地敏感信息，并不会执行什么别的代码。

3.PHP伪协议

基础知识：

allow_url_fopen：是否允许将URL（如http：//或ftp：//）作为文件处理，默认ON。

allow_url_include：是否允许include/require打开URL（如http：//或ftp：//）作为文件处理，PHP5.2以后就变成默认为OFF了。

漏洞利用：直接看参考网站5，用的比较多的就是php://协议和data://协议，常用来读取源码和写入木马的。

PS：正好刷到BUGKU上的web8，点进去给了源码。

预期解是猜测flag.txt的存在，然后得到flag.txt里的内容在GET传参。但是靠猜我觉得不大稳，还看到网上师傅们有非预期解。payload：http://123.206.87.240:8002/web8/?ac=aaa&fn=php://input在POST传参”aaa”就得到flag了。

我这里一开始很不解，因为我去查file_get_contents()这个函数发现如果查找不到文件会返回false，那他是怎么相等的？？之后发现是自己想错了。这里并非是先令fn=POST的值，然后再把这个值作为参数去file_get_contents()。而是先GET传参得到fn=php://input，然后通过file_get_contents(“php://input”）来直接读取POST的数据。可以理解成file_get_contents(“php://input”)是个固定搭配，就是把POST的数据当成一个文件读取的。
