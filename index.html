1.IP地址

IP是一个32位二进制的字符串，比如类似”192.168.70.41”这样的字符串。

IP地址=网络号+主机号。

其中网络号就是前24位二进制，他标志主机所连接到的网络，一个网络号在互联网范围内是唯一的。由IP管理机构分配给单位。路由器仅根据主机所连接的网络号转发分组，不考虑主机号。

而主机号就是后八位二进制，标志该主机或路由器在它所在的网络范围内是唯一的。由得到该网络号的单位自行分配主机号。

在上面的例子中，192.168.70.41(IP地址)=192.168.70.0(网络号)+0.0.0.41(主机号)。

N1CTF Web的总结

其他题完全看不懂。。。就只说SignIn吧

进去直接给源码

问题的关键就在getflag()这个函数，这个函数中只要满足了*md5($this->check)===md5(“key\***************“)**这个条件就ok了。我一开始的思路是想办法绕过md5的验证，但这里用的是===再加上右边恒定是个字符串，真的绕不过去。那么之后考虑怎么才能获得这个key值。可以看到下面是给了一个ip的回显点的，看到上面给的一个类ip中的xff，就想到或许我们可以利用一下ip这个类通过xff注入得到key值。

（一）基础知识

1.这里先说一下xff注入是个什么东西吧。参考网站放在这个题解的最后。

XFF，是X-Forwarded-for的缩写，用来表示 HTTP 请求端真实 IP，而XFF注入是SQL注入的一种，该注入原理是通过修改X-Forwarded-for头对带入系统的dns进行sql注入，从而得到网站的数据库内容。

当然，sql注入很容易想到，xff可以进行union注入，方法也与普通的union注入类似，这里我们主要说xff漏洞注入。xff报错注入利用了updatexml()函数。

关于updatexml()函数：

举个例子，建立了一个user表如图

如果我想知道这个表所在的库，可以构造注入语句：

select name from user where id=1 and updatexml(1,concat('~',(select database()),'~'),3);

就会回显

发现成功得到库名。

原因:由于updatexml的第二个参数需要Xpath格式的字符串，以~开头的内容不是xml格式的语法，concat()函数为字符串连接函数显然不符合规则，但是会将括号内的执行结果以错误的形式报出，这样就可以实现报错注入了。

2.关于stristr()

这个函数是不区分大小写的strstr()，即不区分大小写查询字符串，如果传参是一个实例对象，那就会自动调用__toString先转成字符串。

（二）解题思路

正好__wakeup()里还有这么一句：if(stristr($this->ip, "n1ctf")!==False)可以让我们通过stristr()调用__toString().反序列化this->ip为new ip()，在new ip的时候，成员ip就初始化为xxf~再利用stristr()调用this->ip里的__toString()，然后利用xff报错注入来进行布尔盲注~

得到key值：n1ctf20205bf75ab0a30dfc0c

随后序列化，payload：http://101.32.205.189/?input=O:4:"flag":2:{s:2:"ip";s:5:"xxxxx";s:5:"check";s:25:"n1ctf20205bf75ab0a30dfc0c";}

几个问题：

1.Q:为什么updatexml的第一个和第三个参数是数字1。

A:实际上这两个1是几都没问题，因为updatexml()由于报错，执行只用到了第二个参数也就是我们的concat，而第一个和第三个参数压根没用到。

2.Q:脚本中payload为什么要用f’’格式化？直接改成双引号再把字符串里面的转义符去掉为什么不行(传出来的结果是字典(word))？

A:f-string在本质上并不是字符串常量，而是一个在运行时运算求值的表达式。也就是说，虽然用格式化和用双引号得到的字符串形式上相同，但是本质上后者不会运行里面的语句，而是被当成一个普通的字符串。被当成普通字符串的结果就是肯定最后检测到n1ctf这个子串(自己看payload里是不是出现了n1ctf)，这样就会恒有”welcome to n1ctf”，这样自然的，字典中所有元素都满足条件，就全都出去了/捂脸

3.Q:http://101.32.205.189/?input=O:4:"flag":2:{s:2:"ip";O:2:"ip":0:{};s:5:"check";s:25:"n1ctf20205bf75ab0a30dfc0c";}这个payload为什么没有弹出flag？

A:O(bject)序列化的{}后是没有分号(;)的QAQ，改成http://101.32.205.189/?input=O:4:"flag":2:{s:2:"ip";O:2:"ip":0:{}s:5:"check";s:25:"n1ctf20205bf75ab0a30dfc0c";}就有flag了(该学写脚本了啊)

4.Q:为什么我不直接 select key from n1key 得到key？

A:因为报错信息根本不回显，连查出来的key都是在我们看不见的后端验证的，所以只能布尔盲注一点一点得到key。

5.Q:在网上查substring()说参数中后两个数字是开始和结束。那为什么结束是1？

A:仔细看看这是哪的substring()!网上查到的是java和js的。而这里的substring()是sql server的(即第三个参数不是结束，而是子串长度)，他们的substring()参数并不完全一样。

SQL注入的总结

到目前为止我也做了好多SQL注入的题了，但是每次做都觉得很混乱，整理一下看来是很有必要的。

目前我只接触到了回显注入、堆叠注入和宽字节注入，之后如果有新的再补充

一、常规步骤：

PS:爆库/表/数据前要先确认回显点，并且在回显点查询的信息和数量要和字段数匹配才行

(-1)万能密码：1’ or 1=1#

(0)判断字符型还是数字型

(1)测试字段数

EG:1’ order by 1#

PS:也可以联合查询一点一点试

(2)爆库

联合查询:1’ union select 1,database() #

堆叠注入:1’;show databases;#

(3)爆表

联合查询:1’ union select group_concat(table_name),table_schema from information_schema.tables where table_schema = ‘数据库名’#

学习:

group_concat是一个拼接函数，他在这把所有表名拼在一起，这样回显出来就全都爆出来了

而information_schema是一个mysql自带的表，它里面储存的是“元信息”，可以简单认为，他储存的信息是用来储存信息的，因此要查询储存的表信息当然需要from information_schema.tables。而table_schema更像是一个锁定器，用来锁定要查询的数据库的，他实际上就是表示表所在的数据库的一个参数。

堆叠注入:1‘；show tables;#

联合查询:1’ union select 1,group_concat(column_name) from information_schema.columns where table_name=’表名’ and table_schema=’库名’ #

堆叠注入:1’;show columns from 表名;#

(5)爆出数据

联合查询:1’ union select 列名1, 列名2 from 表名#

二、绕过

很多时候会发现，上面的步骤好像并没有那么顺利。“别黑了”、“stop hacking!”类似这种黑话见到早就不止一次了QAQ。因为现在哪还有那么友好的出题人出题是为了让你按照公式直接秒的，大多时候出题人都设置了白名单过滤，一旦你输入的payload被检测到匹配白名单中的某一项，注入就失败了。下面总结一下见过的白名单过滤以及绕过方法。

(1)过滤关键字

比如过滤select/or/union等等。

绕过方法：

<1>双写绕过

这种后端就是把白名单检测到的东西变成””这样我们双写一次就ok

比如select我们就改成seleselectct，中间加粗的select没了，还剩下一个，完美

但是要注意一个小细节，wyb学长曾经犯过的一个错误被我深深记下，php双写不能双写成phphpp，因为表面看你以为加粗的php没了，但实际上是前面的php没了，也就是过滤后剩下的是hpp，这个地方不要错第二次啊！

<2>大小写绕过

比如select写成SElecT，反正php对大小写不敏感(说php垃圾好像不大合适啊)

<3>过滤函数是十六进制过滤

后台代码可能把关键字给十六进制加密了，我们自己解密一下就好了

select可以写成selec\x74，就类似这样的

<4>双重URL编码绕过

这种情况一般就是说后端检测一次后自己又解密了一次。后台查询的可能是urldecode($_GET[‘id’])。

比如这个题就用到了这个知识点，当然虽然这个题不是注入只是单纯考绕过就是了，这个题详细wp在这。

(2)宽字节注入

过滤原理是利用addslashes函数给输入的字符串中的特殊字符前加上反斜杠（\），让我们想要使用的命令变成单纯的字符串，特殊字符串有单引号（’）、双引号（”）、反斜杠（\）和NULL

绕过原理直接看这里。

刚好做过一个题就是用到宽字节注入的，就在这，忘了的话可以做做回想一下。

(3)过滤空格

<1>注释符过滤

可以用’#’、’–’、’//‘、’/**/‘、‘；%00’来代替空格符

<2>二次URL编码绕过

一次编码”%20“，则可以用二次编码“%2520”尝试绕过

<3>空白字符绕过(十六进制)

比如利用换行符绕过

“select

->user,password

->from

->users”

<4>特殊符号绕过

特殊符号指反引号(`)、加号(+)等

比如构建payload使后端变为”select`user`,`password` from”

<5>科学计数法绕过

PS：md5也有一个0e漏洞，不过和这里的绕过不是同一个东西2333，md5的0e漏洞可以参考这里，md5的0e漏洞目的是为了找到符合”md5值相等但原值不同”的两个变量来满足题目条件，比如满足

“if(md5($key1) == md5($key2) && $key1 !== $key2){
echo $flag.”取得flag”;
}“

，而不是绕过判定

那么我们这里讲的科学计数法绕过是为了什么呢？是为了欺骗白名单过滤，让他以为我们的关键字是数字0，从而让我们的恶意代码顺利进入后端。

比如如果我们使用?id=0e1union select 1,2

后端就变成了 SELECT user,password from users where user_id=0 union select 1,2//0e1变成0后面自动补了个空格为了区分后面的内容，但是白名单过滤的时候是区分不出来的，白名单只会认为0e1union是一个字符串，真正开始查询的时候0e1才会变成0+空格

因此如果白名单中有union，那么通过科学计数法，union就可以顺利绕过过滤了

(1)注入对象构造方法

(2)__wakeup()绕过

反序列化时，如果表示对象属性个数的值大于真实的属性个数时就会跳过wakeup( )的执行。

题目原码发现file是经过base64_decode的，所以必须得把上面的字符串先base64加密。然而，问题出现了。

将上面的payload加密后发现根本读不出flag，去看了大佬的payload解密后发现原来的%00变成了一个莫名其妙的字符，下面是大佬的payload：

O:+4:”Demo”:3:{s:10:”�Demo�file”;s:8:”fl4g.php”;}

对应

TzorNDoiRGVtbyI6Mzp7czoxMDoiAERlbW8AZmlsZSI7czo4OiJmbDRnLnBocCI7fQ==

之后我想明白为什么了。在一般的题中，不牵扯base64加解密，直接在网址中输入序列化的字符串，然而url是可以自动识别出%00也就是截断符的。说白了，%00本身就是没法显示出来的，二进制序列化后，截断符就是长得上图那个样。而如果我先base64加密了，url识别的时候是识别的base64加密的东西，%00识别不出来。等到解密的时候，php这么垃圾的语言根本读不出%00是截断符，只会把它当做’%’’0’’0’三个字符。

那我们咋办啊，之后看了很多大佬的wp豁然开朗。我他妈自己造序列化字符串干啥啊，直接用脚本改就完事了。这里直接上大佬写的脚本，不知道怎么写出能让php认出来的%00，那就不写他，让他自己生成，我把需要改的地方改了不就完事了。
